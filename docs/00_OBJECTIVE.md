# Objetivo del proyecto `l10n-paraguay`

## 1. Misión

Construir el **primer set serio de módulos OCA-style** para la localización fiscal
de **Paraguay** sobre **Odoo Community 18**, cubriendo el ciclo completo de
facturación electrónica DNIT/SIFEN, plan de cuentas paraguayo, reportes fiscales
y POS integrado.

Hoy Paraguay no tiene una localización de calidad publicada en OCA ni en
Enterprise — quienes implementan Odoo en Paraguay arman soluciones cerradas,
fragmentadas, y suelen depender de servicios pagos externos. Este repositorio
busca **cambiar ese estado del arte**: producir módulos OCA-compliant, abiertos
(AGPL-3), reutilizables por toda la comunidad, y con tests/documentación a la
altura de l10n-spain o l10n-brazil.

## 2. Audiencia objetivo

### Inicial (primer año)

- **PyMEs paraguayas con punto de venta físico**:
  - Minimarkets de barrio
  - Comercios pequeños/medianos
  - Gastronomía pequeña (cafeterías, comedores, bares)
- Necesidades típicas: facturación rápida, KuDE imprimible en ticket, cumplir
  SIFEN sin pagar SaaS mensual
- Tamaño: 1-50 empleados, 1-3 puntos de venta

### Secundaria (segundo año)

- Empresas de servicios profesionales (estudios contables, agencias de viajes —
  tipo régimen 8 con IVA al 5%)
- Empresas medianas con e-commerce (módulo `l10n_py_website_sale` futuro)

### Fuera de scope (al menos hoy)

- Grandes empresas con flujos custom enterprise
- Sectores regulados específicos (farmacias con régimen especial, importadores
  con ICE, etc.) — pueden requerir modules adicionales fuera de este repo

## 3. Stack

| Capa           | Tech                                       | Versión  |
| -------------- | ------------------------------------------ | -------- |
| Framework Odoo | **Odoo Community Edition** (NO Enterprise) | **18.0** |
| Lenguaje       | Python                                     | 3.11+    |

| OS soportado   | Linux (target prod) / Windows (target dev) | —        |

### Librerías Python (runtime)

- `lxml` — generación + parsing XML SIFEN
- `signxml` — firma XAdES-BES del DE
- `zeep` — cliente SOAP contra WSDL DNIT
- `cryptography` — manejo PKCS12, RSA, hashing
- `qrcode` — generación QR para KuDE
- `requests` (transitive)
- `requests-pkcs12` — mutual TLS con CCFE para SIFEN

Sin dependencias propietarias. Sin proveedores intermedios (no IAP, no OSE,
conexión directa a SIFEN).

Detalles en [`40_PYTHON_LIBRARIES.md`](40_PYTHON_LIBRARIES.md).

## 5. Principios

### Convenciones de código

- **Idioma del código**: inglés (variables, funciones, clases, modelos, XML IDs)
- **Idioma de comentarios y docstrings**: español
- **Idioma de UI** (strings, labels): español + i18n via `.pot` (luego Weblate
  agrega `es_PY` específico si difiere de `es`)

### Calidad

- **Tests obligatorios**: unit para lógica de dominio, integration para
  adaptadores (SIFEN, KuDE), tour/HTTP para flujos UI relevantes
- **Cobertura objetivo**: 80%+ en módulos nuevos
- **Pre-commit OCA** activo: black, isort, pylint-odoo, flake8, oca-checks-\*
- **CI verde antes de merge** (GitHub Actions con OCA/oca-ci)

### Arquitectura

- **OCA-strict**: naming `l10n_py_*`, AGPL-3, manifest OCA-compliant
  (`author` termina en `, Odoo Community Association (OCA)`)
- **Modelos delgados + servicios puros**: lógica compleja (CDC, firma, XML
  building) en módulos Python puros bajo `services/`, testables sin levantar
  Odoo
- **Reutilizar core**: `account.move`, `res.partner`, `account.tax`,
  `account.edi.document`, `l10n_latam_invoice_document` — no reinventar
- **Sin "modelos paralelos"**: no crear `l10n_py.invoice` cuando `account.move`
  ya hace el trabajo (rompe integración con sale/purchase/pos/stock)

### Documentación

- **Toda la documentación técnica vive en `docs/`** — `CLAUDE.md` solo es
  manifiesto + tabla de contenido
- Cada doc tiene **frontmatter YAML** (source, fetched_at, summary, priority)
- README por módulo (cuando exista) generado por `oca-gen-addon-readme` desde
  `readme/` fragments
- Cambios de arquitectura **se reflejan en `docs/`** en el mismo PR

### Git workflow

- **Conventional Commits** (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`)
- Branching: `feat/<name>`, `fix/<name>`, merge a `main` vía squash
- Pre-commit hooks bloquean commits con errores de lint
- PRs requieren review antes de merge

## 7. No-objetivos (lo que explícitamente NO hacemos)

- **NO** desarrollar un fork de Odoo
- **NO** crear módulos para Odoo Enterprise (puede vivir aparte si surge demanda
  pagada en el futuro)
- **NO** ofrecer SaaS — el software es self-hosted o gestionado por terceros
- **NO** competir con TIPS-SA u otros providers de librerías SIFEN — sumarlos
  como dependencias cuando aplique (en Python si existieran)
- **NO** rehacer features que Odoo core ya provee (POS, sale, purchase, stock,
  account moves, attachments, mail)
- **NO** asumir conexión permanente a internet — el flujo SIFEN tolera offline
  hasta 72h (el framework `account.edi.document` ya lo maneja)
- **NO** soportar versiones de Odoo previas a 18.0 inicialmente — backports
  pueden surgir si OCA los pide pero no son prioridad

## 9. Relación con ÑandeFact (proyecto previo)

`l10n-paraguay` **NO** es un port literal de ÑandeFact a Odoo. ÑandeFact era una
solución vertical (app Android + backend Node) optimizada para un nicho
específico (vendedoras de mercados populares con devices low-end). Este proyecto
es **horizontal**: módulos genéricos OCA-style usables por cualquier PyME.

Lo que sí se hereda:

- **Conocimiento de dominio SIFEN** (CDC, XML, web services, eventos) — en
  [`01_SIFEN_KNOWLEDGE_BASE.md`](01_SIFEN_KNOWLEDGE_BASE.md) y
  [`02_SIFEN_REFERENCIA_COMPLETA.md`](02_SIFEN_REFERENCIA_COMPLETA.md)
- **Patrones de UX** validados en mercados (3 taps facturación, KuDE en ticket,
  consumidor final por default) — aplicables al diseño de `l10n_py_pos`
- **Modelo conceptual** de invariantes y validaciones — en
  [`03_DOMAIN_MODEL.md`](03_DOMAIN_MODEL.md)

Lo que NO se hereda:

- Stack (Node/TS → Python)
- Arquitectura hexagonal estricta (Odoo idioma)
- Optimizaciones offline-first (Odoo soporta pero no es target inicial)
- Frontend Android (futuro `l10n_py_pos` mobile podría revivir esto)
