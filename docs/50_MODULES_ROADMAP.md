## Fase 1 — `l10n_py_base` + `l10n_py_account`

**Duración estimada:** 4-6 semanas.

### Scope
- `l10n_py_base`
  - Catálogos DNIT/SIFEN: departamentos (17), distritos (~258), ciudades (~3400),
    actividades económicas, tipos de régimen
  - Modelo `l10n_py.timbrado`
  - Modelo `l10n_py.point_of_emission`
  - Extensión `res.company`: RUC + DV, nombre fantasía, tipo contribuyente,
    régimen, actividades
  - Extensión `res.partner`: identification types Paraguay (records de
    `l10n_latam.identification.type`), validación CI/RUC
  - Wizard de upload de CCFE (encriptación AES) — opcional acá, puede ir a
    `l10n_py_edi`
- `l10n_py_account`
  - Chart of accounts paraguayo (investigar si hay un estándar OCA-friendly)
  - Taxes (IVA 10%, 5%, exenta, 0% exportación)
  - Records de `l10n_latam.document.type` (FE 01, AF 04, NC 05, ND 06, NR 07)
  - Extensión `account.journal`: campos timbrado + point of emission
  - Configuración default de `l10n_latam_use_documents = True` en sale journals

### Dependencias
- `account`
- `l10n_latam_base`
- `l10n_latam_invoice_document`

### Validación (Definition of Done)
- [ ] Módulo se instala sin warnings en una DB limpia con `-i l10n_py_account`
- [ ] Demo data: company de prueba con RUC válido, partner con CI válido,
      producto con tax 10%
- [ ] Tests: validación RUC, validación CI, single-active-timbrado constraint
- [ ] Pre-commit limpio
- [ ] Document types aparecen automáticamente al crear `account.move` con
      `move_type='out_invoice'` en company PY

### Riesgos
- **Chart of accounts**: si no hay un PUC estándar paraguayo accesible, hay que
  construirlo desde cero o adaptar uno latinoamericano. Investigar con contadores
  locales (Ez probablemente tiene contacto).
- **Catálogos completos** (3400+ ciudades): obtener como CSV oficial vs scraping.
  Idealmente downloads del portal e-Kuatia.

## Fase 2 — `l10n_py_edi` (MVP: FE + NC + KuDE)

**Duración estimada:** 8-12 semanas.

### Scope MVP
- Record `account.edi.format` con code `'py_dnit_sifen'`
- Subclase Python con `_get_move_applicability`, `_post_invoice_edi`,
  `_cancel_invoice_edi`
- Generador XML del DE (`services/xml_builder.py`) — grupos AA/A/B/C/D/E/F/H
  para FE y NC
- Firmador XAdES (`services/xmldsig.py`) — usa signxml + PKCS#12
- Cliente SOAP SIFEN (`services/sifen_client.py`) — siRecepDE, siRecepLoteDE,
  siConsDE, siResultLoteDE
- Cliente eventos (`services/sifen_event_client.py`) — siRecepEvento
- Generador CDC (`services/cdc.py`) con validación módulo 11
- Wizard `l10n.py.cancellation.wizard` — UI para cancelar DTE
- Wizard `l10n.py.inutilizacion.wizard` — UI para inutilizar rango
- Reporte QWeb `kude_template.xml` — A4 (cliente) y formato ticket (POS)
- Encriptación CCFE en BD (Fernet con key en `ir.config_parameter` o env var)
- Cron de consulta de lotes pendientes
- Logs en chatter de cada move

### Dependencias
- `l10n_py_account`
- `account_edi`
- (`account_edi_ubl_cii` — solo si compartimos algún builder UBL, probablemente no)

### Validación
- [ ] Cargar CCFE de prueba (provista por DNIT en homologación) sin errores
- [ ] Generar XML que pase validación XSD oficial offline
- [ ] Firma del XML pasa verificación con OpenSSL externo
- [ ] Enviar a `sifen-test.set.gov.py` y recibir `0260` para una factura simple
- [ ] Enviar lote de 5 facturas, consultar resultado, todas aprobadas
- [ ] Cancelar una DTE aprobada vía wizard, recibir aprobación de evento
- [ ] KuDE PDF renderiza correctamente con QR escaneable
- [ ] Inutilizar un rango vía wizard, recibir aprobación
- [ ] Tests con mocks: 100% del flujo verde sin SIFEN real
- [ ] Tests `external`: 100% verde contra SIFEN test

### Riesgos
- **Homologación SIFEN puede tardar**: depende de obtener CCFE de prueba (un
  cliente real con timbrado test, o solicitar Ez como persona física)
- **Firma XAdES**: SIFEN es estricto con canonicalization. Puede requerir
  iteración para que la firma sea aceptada
- **TLS mutual con PKCS#12 en Python 3.13**: hay edge cases — tener fallback
  plan a conversión .p12 → .pem si requests-pkcs12 da problemas

### Métrica final
> Una FE creada en Odoo PY, postear, esperar cron, ver `edi_state='sent'` con
> XML firmado en attachments y respuesta SIFEN en chatter. Generar KuDE PDF que
> al escanear QR lleva a e-Kuatia y muestra el DTE aprobado.

## Fase 3 — `l10n_py_reports` (Libros + Hechauka + DDJJ)

**Duración estimada:** 6-8 semanas.

### Scope
- **Libro IVA Ventas**: lista DTEs `out_invoice` + `out_refund` del periodo con
  totales por tasa
- **Libro IVA Compras**: lista DTEs `in_invoice` + `in_refund` con totales y
  retenciones aplicadas
- **Hechauka**: estructura específica DNIT (formato CSV/TXT). Investigar última
  versión del manual
- **DDJJ IVA mensual**: Form 120 (o equivalente vigente)
- **DDJJ IRE/IRP**: si community Odoo tiene framework de reports adecuado
  (verificar `account_reports` no-enterprise alternative)

### Dependencias
- `l10n_py_edi`
- `account` (reports framework)
- Probable: `account_reports` (Enterprise) o un OCA alternativo

### Validación
- [ ] Libro IVA Ventas/Compras cuadran con suma de moves de la company
- [ ] Hechauka exporta TXT que pasa validación del DNIT en su sistema
- [ ] DDJJ IVA refleja correctamente los créditos y débitos del periodo

### Riesgos
- Si `account_reports` Enterprise no se puede usar, hay que escribir reports
  custom (más laborioso pero factible)
- Formato Hechauka cambia con frecuencia — versionar el exporter

## Fase 4 — `l10n_py_pos`

**Duración estimada:** 6-8 semanas.

### Scope
- Extensión `pos.order` y `pos.session` para integración SIFEN
- Al cerrar sesión POS: las pos.orders generan account.moves; cada move dispara
  flujo EDI normal
- Modo "factura inmediata" desde POS (usuario seleccionando partner con CI/RUC
  fuerza envío SIFEN ya, no espera cierre)
- KuDE en formato ticket impreso (con QR)
- Soporte "consumidor final" (innominado) hasta monto límite
- Validación de partner: CI o RUC obligatorio si monto > límite
- Frontend POS (OWL components) extendido para mostrar campos SIFEN

### Dependencias
- `l10n_py_edi`
- `point_of_sale`

### Validación
- [ ] Vender desde POS sin partner (innominado) → factura con tipo_documento=5
      en cliente
- [ ] Vender con partner identificado (CI) → factura con datos del cliente
- [ ] KuDE imprime en impresora térmica (test con XPrint o similar)
- [ ] Sesión POS con 50+ ventas → todas envían a SIFEN al cierre, sin warnings
- [ ] Performance: POS responsive (<200ms por venta) aunque haya backlog SIFEN

### Riesgos
- **Performance del POS**: el cálculo del CDC al cerrar la sesión no debe
  bloquear la UI — usar background job
- **Impresoras térmicas variadas**: testear con al menos 2 modelos comunes en
  PY (Epson, XPrint, Star)
- **Modo offline POS**: Odoo POS tiene offline mode nativo; verificar que la
  generación del CDC al recuperar conexión funciona

## Fase 5 — `l10n_py_withholding`

**Duración estimada:** 4-6 semanas.

### Scope
- **Retención IVA**: % retenido sobre IVA en compras (vendor bills)
- **Retención IRE** (Impuesto a la Renta Empresarial): % sobre base imponible
- **Retención IRP** (Impuesto a la Renta Personal): % sobre base imponible
- Configurable por:
  - Tipo de partner (agente retenedor declarado)
  - Tipo de producto/servicio
  - Monto del comprobante (escalas progresivas si aplica)
- Generación de "Comprobante de Retención" — formato DNIT específico
- Reporte de retenciones mensuales/anuales

### Dependencias
- `l10n_py_edi` (para flujo EDI si el comprobante de retención también es DTE)
- `l10n_py_reports` (para reportes de retenciones)
- Inspirar en `l10n_ec_withhold` (OCA Ecuador) y `l10n_ar_withholding` (Odoo)

### Validación
- [ ] Crear vendor bill con tax retención IVA → vendor payment registra el
      withholding
- [ ] Comprobante de retención generado con datos correctos
- [ ] Reporte mensual de retenciones cuadra

### Riesgos
- Definición legal de retenciones puede cambiar (decretos DNIT) — desacoplar
  cálculo del código (usar configuration models, no constantes)

## Reglas transversales para todas las fases

1. **No mergear sin tests** — feature en draft sin tests permitido, merge no.
2. **Documentación viva** — cualquier cambio de arquitectura → update en `docs/`
   en el mismo PR.
3. **No backports a Odoo <18.0** hasta que el módulo esté estable y haya demanda.
4. **No depender de Enterprise** — si una feature requiere algo Enterprise,
   documentar el workaround community o saltarla.
5. **Tags consistentes para tests externos**: `@tagged('-standard', 'external', 'l10n_py')`
   para todo lo que toca SIFEN real.
6. **Versiones semver-Odoo correctas**: bump `y` al agregar feature, `z` al
   fixear, `x` al cambiar schema.
